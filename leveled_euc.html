Leveled - The Hypothesis

Disk I/O is an unpredictable bottleneck

Be non-generic, focus on specific context: e.g. larger objects; many stores per server

Riak doesn't always need to know the value

Leveled - The Key Changes

Make the nursery log a permanent journal of key and values

Roll only the Keys and Metadata into the merge tree

In-memory state in small buffers, bloom filters and fadvise

Support HEAD requests, and metadata folds

Objects can be PUT with TAGs where TAGs define behaviour

Leveled - The Actors

Bookie - marshalls requests at the front

Inker - keeps permanent journal of keys and values

Penciller - manages merge tree (ledger) of keys and metadata

Clerks - support Inker and Penciller with background tasks

Leveled - The Inker and the Journal

The journal is a series of (DJ Bernstein) CDB files (as FSMs)

The Inker has a manifest of sequence numbers -> FSMs -> Files

The Inker has a Clerk to help with compaction

Active journal file is mutable by append

Other journal files are immutable until replacement

Leveled - The Penciller and the Ledger

The Bookie pushes lists of key/metadata in chunks

The Penciller stacks recent chunks in memory, with hash-based index

When in-memory count exceeds size of a sst file - persist and flush

Each level consists of immutable SST files in key order

Each level has a maximum number of files - excess triggers merge

Leveled - Operations

PUT - Inker commits to Journal, Bookie caches change to Ledger

GET - Penciller fetches SQN, Inker fetches value

HEAD - Penciller fetches metadata from Ledger

INDEX - Additional key/metadata changes in Ledger

FOLD - Efficient in key-ordered ledger through clones of Penciller

CLONE - By manifest copy, with delete_pending file state, allowing reads in parallel

Leveled - Volume tests

Significant throughput improvements where disk I/O is the dominant constraint

  • With sync enabled (flushing each and every write)
  • With spinning disk drives not solid-state drives

Focused on testing without sync on SSD since

  • Throughput advantage at > 4KB values
  • Advantage increases with value size
  • Lower mean PUT times, higher median GET times
  • Dramatic reduction in tail latency and volatility
